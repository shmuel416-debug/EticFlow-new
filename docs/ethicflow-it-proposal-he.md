## שקופית 6 — למה Cloud ולא On-Premise?

**שקלתי ארבע אפשרויות:**

### א. On-Premise (שרתים פיזיים של המוסד)
- ✅ שליטה מלאה
- ❌ דרושים שרתים, רישיונות, תחזוקה, גיבויים
- ❌ זמן הקמה: 2-3 חודשים
- ❌ עלות התחלה: ~$5,000-10,000 + תחזוקה שוטפת

### ב. שרת וירטואלי (VM) של המוסד
- ✅ ניצול תשתית קיימת
- ❌ דורש ידע DevOps רחב
- ❌ אחריות תפעולית מלאה עליי או על IT
- ❌ אין auto-scaling, אין HA out-of-the-box

### ג. Cloud Public (Azure / AWS / GCP) — **המומלץ**
- ✅ services מנוהלים (DB, Storage, Auth)
- ✅ סקלביליות אוטומטית
- ✅ אבטחת מידע ברמת ספק
- ✅ זמן הקמה: 1-2 שבועות
- ❌ עלות חודשית קבועה

### ד. SaaS צד שלישי (Railway/Vercel/Render)
- ✅ זול ($15-30/חודש)
- ❌ **לא תואם** לדרישות מוסד אקדמי (data residency, ISO 27001)
- ❌ ועדת אבטחת מידע לא תאשר

**ההמלצה: Azure Public Cloud בתשתית המוסד.**

## שקופית 7 — למה Azure ולא AWS/GCP?

**שלוש סיבות עיקריות:**

### 1. אינטגרציה טבעית עם Microsoft 365 של המוסד
המערכת תומכת ב:
- **Entra ID SSO** — משתמשי המוסד נכנסים בלי סיסמה נוספת
- **Microsoft Graph Mail** — שליחת הודעות מתיבת המוסד
- **Outlook Calendar** — סנכרון ישיבות הוועדה ליומן
- **OneDrive/SharePoint** — אופציונלי, לאחסון מסמכים

ב-AWS/GCP אינטגרציות אלה הן workarounds. ב-Azure הן native.

### 2. תאימות וביקורת
Azure מספק **out-of-the-box**:
- ISO 27001, ISO 27017, ISO 27018
- SOC 1, 2, 3
- GDPR compliance
- Data residency באירופה (West Europe = הולנד)
- BAA חתום

### 3. ידע ארגוני קיים
המוסד כבר עובד עם Microsoft (Office 365, Azure DevOps). אם בעתיד מצטרפים מפתחים נוספים — הם מכירים את הסביבה.

## שקופית 11 — תהליך Deploy ושחרור גרסה

```
מפתח עובד מקומית
       ↓
    git push בעניף feature
       ↓
   Pull Request לבחינת קוד
       ↓
   Azure Pipelines CI:
   ✓ lint, tests, security scan
       ↓
   Merge ל-develop branch
       ↓
   ─────── Auto Deploy ל-DEV ───────
       ↓
   בדיקה ידנית בסביבת Dev
       ↓
   Pull Request מ-develop ל-main
       ↓
   ─────── Manual Approval Gate ─────
   • Wait timer: 5 דקות
   • Required Reviewer: מנהל המערכת
   • CI חייב לעבור
       ↓
   ─────── Deploy ל-PRODUCTION ──────
       ↓
   Health Check אוטומטי על /api/health
       ↓
   אם נכשל → Rollback אוטומטי לגרסה הקודמת
       ↓
   Notification ל-IT: "Deploy הושלם בהצלחה"
```

**Downtime במהלך deploy:** ~30 שניות (B1 plan).
**אם המוסד דורש 0 downtime:** שדרוג ל-S1 plan (+$60/חודש) → slot swap = 0 downtime.

## שקופית 12 — אבטחת מידע ופרטיות

**עקרונות יישום:**

### בקרת גישה
- **Microsoft Entra ID SSO** — חיבור לזהות הארגונית. אין סיסמאות נפרדות.
- **JWT** — סשן אבטחתי, expiration אוטומטי
- **RBAC** — חמישה תפקידים מוגדרים, הרשאות לפי תפקיד
- **Audit Log** — כל פעולה רגישה מתועדת (who, what, when)

### הגנה על נתונים
- **בתעבורה**: HTTPS בלבד (TLS 1.2+, HSTS)
- **במנוחה**: Azure Storage Service Encryption (AES-256)
- **DB**: Transparent Data Encryption (TDE) של Azure
- **סודות**: Azure Key Vault — לא ב-code, לא ב-env files

### עמידה בתקנים
| תקן | סטטוס |
|------|-------|
| OWASP Top 10 | ✅ 100% (security audit ב-Sprint 6) |
| IS 5568 (נגישות) | ✅ מלא |
| WCAG 2.2 AA | ✅ מלא |
| חוק הגנת הפרטיות התשמ"א-1981 | ✅ מותאם |
| GDPR | ✅ עקרונות מיושמים |
| ISO 27001 | ✅ דרך Azure |

### תרחישי איום שטופלו
- SQL Injection (Prisma ORM = אין raw SQL)
- XSS (sanitization בכל קלט)
- CSRF (token validation)
- Brute force (rate limiting)
- Session hijacking (JWT signed + short expiration)

## שקופית 13 — אינטגרציה עם תשתיות המוסד

**שלוש אינטגרציות מוכנות (קוד כבר נכתב, חסרה רק תצורה):**

### 1. Microsoft Entra ID — SSO
משתמשי המוסד נכנסים עם החשבון הארגוני שלהם. אין סיסמה נפרדת.
- צריך מ-IT: App Registration ב-Entra ID, Client ID + Client Secret
- מקסימום עבודה צפויה ב-IT: שעה

### 2. Microsoft Graph — Email
המערכת שולחת התראות (אישורי ועדה, דחיות, תזכורות SLA) דרך תיבת מייל ייעודית של המוסד.
- צריך מ-IT: תיבת מייל (no-reply@... או ethics-committee@...), הרשאות Mail.Send על Service Principal
- מקסימום עבודה צפויה ב-IT: שעה

### 3. Microsoft Outlook Calendar
ישיבות הוועדה מסונכרנות ליומן Outlook. כל חבר ועדה מקבל הזמנה אוטומטית.
- צריך מ-IT: הרשאות Calendars.ReadWrite על Service Principal
- מקסימום עבודה צפויה ב-IT: 30 דקות

**אופציונליות (לא חובה ב-V1):**
- שילוב עם SharePoint לאחסון מסמכים גדולים
- שילוב עם Power BI לדשבורד מנהלים
- שילוב עם Teams להתראות

## שקופית 19 — שאלות לדיון

### שאלות לאישור עקרוני
1. **תקציב** — האם $100-140/חודש מקובל? יש תקציב ייעודי או ישובץ בתקציב מחלקה?
2. **אזור גיאוגרפי** — האם West Europe (אמסטרדם) מקובל, או יש דרישת data residency בישראל?
3. **תהליך אישור** — האם יש ועדת אבטחת מידע שצריך להציג בפניה? פגישה נוספת?

### שאלות תפעוליות
4. **Domain** — מה ה-subdomain המוצע? `ethics.[university].ac.il`? אחר?
5. **SSL Certificate** — Azure Managed Certificate (חינם), או חובה דרך CA המוסד?
6. **תיבת מייל** — האם נוצור תיבה ייעודית או נשתמש בקיימת?

### שאלות אבטחה
7. **Private Endpoints** — האם המוסד דורש שה-DB יהיה לא חשוף לאינטרנט (private only)?
8. **WAF** — האם דרישה מ-IT, או אופציונלי בשלב ראשון?
9. **Penetration test** — האם המוסד מספק או צריך לקנות חיצונית?

### שאלות אינטגרציה
10. **Entra ID** — מי יוצר את ה-App Registration? אני בעצמי או IT?
11. **Microsoft Graph permissions** — תהליך אישור? Admin consent דרוש?
12. **Tenant restriction** — האם המערכת תהיה זמינה רק למשתמשי המוסד או גם לחיצוניים (משתפי פעולה במחקר)?

### שאלות תפעול
13. **תמיכת חירום** — מי בכוננות ב-3 לפנות בוקר? אני, IT, או מוקד?
14. **SLA** — מה ה-uptime היעד? 99%, 99.5%, 99.9%?
15. **תהליך release** — האם נדרש Change Management ארגוני לפני כל deploy?

### שאלות אסטרטגיות
16. **גדילה** — האם בעתיד המוסד שוקל לחלוק את המערכת עם מוסדות אחרים? (משפיע על ארכיטקטורה)
17. **רישוי** — האם המערכת תיחשב כקניין רוחני של המוסד? יש מסמך רשמי?
18. **מסחור עתידי** — אם המערכת תוכח מוצלחת, האם המוסד שוקל הפיכתה למוצר?

## נספח ב — Glossary למקבלי החלטות

| מונח | הסבר בעברית פשוטה |
|------|---------------------|
| **Azure** | פלטפורמת ענן של Microsoft |
| **App Service** | שירות מנוהל להרצת אפליקציות web |
| **Resource Group** | אוסף לוגי של משאבים בענן |
| **Subscription** | יחידת חיוב בענן |
| **Key Vault** | כספת דיגיטלית לסודות (סיסמאות, מפתחות API) |
| **Entra ID** | מערכת זהויות של Microsoft (השם החדש של Azure AD) |
| **SSO** | Single Sign-On — כניסה אחת לכל השירותים |
| **CI/CD** | תהליך אוטומטי של בדיקה והעלאת קוד |
| **SLA** | התחייבות לזמינות (כמה אחוזים מהזמן המערכת תהיה זמינה) |
| **PITR** | Point-in-Time Restore — שחזור מסד הנתונים לרגע מסוים בעבר |
| **WAF** | Web Application Firewall — הגנה מפני התקפות web |
| **Service Principal** | זהות לאפליקציה (לא משתמש אנושי) |
| **Pipeline** | תהליך אוטומטי של בנייה ופריסה |
